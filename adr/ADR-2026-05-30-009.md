# ADR-2026-05-30-009: t09_leaf_signing_gap — pac-ret leaf-function signing gap

**Date**: 2026-05-30
**Status**: Implemented (2026-05-31)
**Author**: Security Architect
**Commit**: 327b741

## Context

LLVM's AArch64 backend, in `/Users/kejor/ee202c/arm-toolchain/llvm/lib/Target/AArch64/AArch64MachineFunctionInfo.h`, defines:

```cpp
enum class SignReturnAddress { None, NonLeaf, All };
```

The default for `-mbranch-protection=pac-ret` is `SignReturnAddress::NonLeaf`. The helper

```cpp
bool shouldSignReturnAddress(SignReturnAddress Cond, bool IsLeaf)
```

returns `false` when `Cond == NonLeaf && IsLeaf == true`. In
`/Users/kejor/ee202c/arm-toolchain/llvm/lib/Target/AArch64/AArch64PointerAuth.cpp`,
`signLR()` and the `PAUTH_PROLOGUE` / `PAUTH_EPILOGUE` pseudos are only
inserted when `shouldSignReturnAddress` returns `true`. As a direct
consequence, leaf functions compiled under `-mbranch-protection=pac-ret`
contain no `paciasp` in the prologue and no `autiasp` in the epilogue.

This produces an attacker-exploitable surface that the current suite does
not exercise: any leaf function whose saved-LR / live-LR can be corrupted
(via a memory write primitive into the stack slot, via inline-asm or
compiler-spill anomalies that mutate X30, or via a sibling-call thunk
that leaks an attacker-controlled value into X30 before the `ret`) will
return to the corrupted target without any authentication check.

The mitigation is `-mbranch-protection=pac-ret+leaf`, which selects
`SignReturnAddress::All`. Under `All`, `shouldSignReturnAddress` returns
`true` for leaf and non-leaf functions alike, so `paciasp` / `autiasp`
are emitted everywhere. This is documented in the ARM ACLE and is the
configuration shipped by some downstream distributions (e.g. Android's
hardened builds) precisely to close this gap.

## Decision

Add a new test case `t09_leaf_signing_gap` to the PAC vulnerability
suite under `/Users/kejor/ee202c/project/tests/pac-vulnerabilities/`.
Unlike t01–t08 (two variants each: `nopac`, `pac`), t09 has **three**
build variants:

| Variant   | Compile flags                                    | Expected outcome |
|-----------|--------------------------------------------------|------------------|
| `nopac`   | `-O0 -g -static -no-pie -fno-stack-protector`    | `PWNED:t09`, exit 0 |
| `pac`     | …above + `-mbranch-protection=pac-ret`           | `PWNED:t09`, exit 0 |
| `pac_leaf`| …above + `-mbranch-protection=pac-ret+leaf`      | Blocked (SIGSEGV/SIGBUS/SIGILL) |

The `pac` variant succeeding (PWN) is the load-bearing observation of
this test. It demonstrates that LLVM's default `pac-ret` policy leaves
leaf functions unsigned. The `pac_leaf` variant trapping demonstrates
that `pac-ret+leaf` closes the gap. The `nopac` variant is the baseline
control identical in spirit to t01–t04's `nopac`.

This is the third test in the suite (after t06 and t08) where a build
variant compiled with PAC enabled is *expected* to PWN, and the first
test in the suite where the expected outcome depends on a finer-grained
PAC policy (`pac-ret` vs `pac-ret+leaf`).

## Rationale

**Why a dedicated test instead of folding into t01.** t01 corrupts the
saved LR of a non-leaf function (`victim` calls `inner_victim`). The
LLVM gap is specifically about *leaf* functions; reusing t01's
two-function structure would not exercise it. A separate test keeps the
demonstration crisp: one source file, one leaf function, one corruption
primitive, one observed outcome per variant.

**Why corrupt X30 directly via inline asm instead of via a stack
overflow.** A leaf function on AArch64 at `-O0` typically does not
spill X30 to the stack (leaf is defined precisely by "makes no calls",
and X30 only needs preservation across calls). Constructing a buffer
overflow that reaches a saved X30 would require forcing the leaf to
spill, which means giving it a call, which would make it non-leaf and
defeat the purpose. A direct `mov x30, %0` inline-asm sequence inside
the leaf function is the cleanest minimal model of an attacker who has
achieved arbitrary write into the LR register (e.g. via a corrupted
register-restore gadget, via a sibling-tail-call into attacker data,
via a compiler-spill bug). The redirect target is
`exploit_success_t09`, which prints `PWNED:t09` and `_exit(0)`s.

**Why a third variant rather than overloading the existing `pac`
column.** Conflating "pac-ret" and "pac-ret+leaf" into a single column
would erase the distinction this test exists to measure. Three variants
preserve the asymmetry as data the harness reports per binary.

**Alternatives rejected:**
- *Use `__attribute__((target("branch-protection=pac-ret+leaf")))` to
  switch policy per-function.* GCC does not support per-function branch
  protection in the form we need on aarch64-unknown-linux-gnu-gcc; the
  attribute exists in newer Clang but the project pins GCC. Per-binary
  flags via `BUILD_RULE` are uniform with the rest of the suite.
- *Make t09 a 2-variant test (`pac_default` vs `pac_leaf`) and drop
  `nopac`.* The `nopac` variant has very low marginal cost (one extra
  binary) and gives the harness a sanity-check baseline: it must PWN.
  Skipping it would lose that sanity check and break the "every test
  has a no-PAC control" invariant of the suite.

## Implementation Notes

### Files to create

- `tests/pac-vulnerabilities/t09_leaf_signing_gap/vuln.c`
  - One leaf function `leaf_victim()` that takes no arguments, returns
    `void`, and does no calls. Its body must contain exactly the
    inline-asm sequence:

    ```c
    asm volatile("mov x30, %0" :: "r"(exploit_success_t09) : "x30");
    ```

    followed by an explicit `return;`. The compiler will lower the
    function's epilogue to `ret`, which under `nopac` and `pac` jumps
    to the now-corrupted X30 (= `exploit_success_t09`). Under
    `pac_leaf` the epilogue is `autiasp; ret`, and `autiasp` poisons
    the corrupted X30 so the subsequent `ret` faults.

  - `main()` calls `leaf_victim()` directly. After return from
    `leaf_victim` (which only happens on `pac_leaf` *if* the auth
    somehow succeeded, which it should not), main should print
    `UNEXPECTED:t09 leaf_victim returned\n` and `return 1;` so the
    harness flags any silent-pass condition as ERROR.

  - Reference includes: `"win.h"` (for `exploit_success_t09`),
    `<stdio.h>`.

  - Style: mirror `t08_pac_oracle_bruteforce/vuln.c`'s file-header
    block (purpose, why the variant outcomes are what they are, the
    LLVM source citation summarized in one paragraph, and the
    "Hardware caveat" pattern — note that the `pac_leaf` variant
    requires real ARMv8.3 PAC at runtime; on hardware without it,
    `autiasp` is a HINT-NOP and the test would PWN under `pac_leaf`
    for the wrong reason. Same Apple-Virtualization / Docker
    linux/arm64 environment caveat as t07 and t08).

  - Important attribute: the leaf function MUST be flagged with
    `__attribute__((noinline))`. Without it `-O0` is usually enough
    to keep it as a real call, but the suite has been bitten by
    optimization-invariance assumptions before; mark it explicitly.
    Do NOT add `__attribute__((noreturn))` — the function must have
    a real epilogue to demonstrate the autiasp/ret gating.

- (No new build-rule file; the existing parametric `BUILD_RULE` in the
  Makefile handles the new test once the variant list is extended.)

7. Update `verify-static`. The current loop iterates over `pac/`
   binaries and requires at least one `paciasp|pacia` and one
   `autiasp|autia`. Extend it to also iterate over `pac_leaf/`
   binaries with the same requirement. In addition, add a t09-specific
   assertion: `pac/t09_leaf_signing_gap` must contain **zero**
   `paciasp` instructions (it is the leaf function — under pac-ret it
   must NOT be signed; that is the whole point), and
   `pac_leaf/t09_leaf_signing_gap` must contain at least one
   `paciasp`. This is the static counterpart of the runtime
   expectation. Implementation sketch:

   ```make
   verify-static: build
       @# existing pac/ check loop here
       @# new pac_leaf/ check loop here
       @# new t09-specific assertion:
       @leaf_pac_count=$$($(OBJDUMP) -d $(BUILD_DIR)/pac/t09_leaf_signing_gap \
           | awk '/<leaf_victim>:/,/^$$/' | grep -cE "paciasp"); \
       if [ "$$leaf_pac_count" -ne 0 ]; then \
           echo "FAIL: pac/t09 leaf_victim contains $$leaf_pac_count paciasp; expected 0" >&2; \
           exit 1; \
       fi; \
       leaf_pac_count_l=$$($(OBJDUMP) -d $(BUILD_DIR)/pac_leaf/t09_leaf_signing_gap \
           | awk '/<leaf_victim>:/,/^$$/' | grep -cE "paciasp"); \
       if [ "$$leaf_pac_count_l" -lt 1 ]; then \
           echo "FAIL: pac_leaf/t09 leaf_victim missing paciasp" >&2; \
           exit 1; \
       fi
   ```

   This is the load-bearing static observation of the entire test —
   the unsigned leaf prologue under `pac` is exactly the gap that
   `pac_leaf` closes.

### Security considerations

- **No new attack surface introduced.** The inline-asm `mov x30, %0`
  is a self-contained corruption of the test process's own X30; it
  does not touch other processes, the kernel, or any external service.
  The win target `exploit_success_t09` only calls `printf` and
  `_exit(0)`, identical to the pattern used by every other test win
  function.
- **Hardware caveat (same as t07, t08).** On a host without true
  ARMv8.3-A PAC, `autiasp` decodes as a HINT NOP. Under such a host
  the `pac_leaf` variant would PWN for the wrong reason (no trap on
  corrupted X30 because no real authentication happens). The project
  runs on Apple Silicon via Docker `linux/arm64`, where real PAC is
  exposed, so this is a documented caveat rather than a blocker. The
  `verify-static` paciasp check on `pac_leaf/t09` gives a
  toolchain-level guarantee that the prologue *was emitted*, even if
  runtime hardware support varies.
- **The PWN-under-pac variant is a feature, not a bug.** The whole
  point of this test is that `-mbranch-protection=pac-ret` does NOT
  block this primitive. A future maintainer who sees `pac` PWNing and
  "fixes" it to trap would be erasing the very signal this test
  exists to surface. The expectations.yml comment and the
  ADR-2026-05-30-009 reference in the README are the safeguards.

### Sequencing / dependencies

- This ADR touches the Makefile's variant model and the harness's
  variant iteration. Both are localized changes — the existing
  `BUILD_RULE` macro and `expectations.yml` parser already accept the
  needed shape, so the diff should be small. No ADR work upstream of
  this is required.
- Recommend implementing in this order to keep verify-static green
  throughout: (1) add `win.h` / `win.c` entries, (2) add `vuln.c`,
  (3) add Makefile variant-list and rules, (4) update

  (7) update verify-static assertions last.

## Consequences

**Enabled:**
- The suite now documents — with a runnable, traceable demonstration —
  that LLVM's default `pac-ret` policy is *not* equivalent to "PAC
  protects every return". Anyone reading the suite output sees a
  concrete pac-variant PWN that is closed by a specific compiler flag.
- The suite gains a per-test variant model. Future tests that need a
  third (or fourth) build configuration can reuse the
  `VARIANTS_<test>` pattern without further Makefile surgery.
- `verify-static` now asserts the *absence* of a PAC instruction in a
  specific function, not just its presence in the binary. This is a
  stronger and more targeted check.

**Constraints introduced:**
- The harness's variant set is no longer a module-level constant; it
  must be derived from `expectations.yml` per test. Any future tooling
  that relied on `from run_tests import VARIANTS` would break (no
  current callers).
- `pac_leaf` adds one extra binary (~3-5 % marginal build time). The
  `pac_leaf` variant of t01–t08 is intentionally NOT built; if a
  future ADR wants to compare `pac` vs `pac_leaf` for those tests it
  must opt them in explicitly via `VARIANTS_<test>`.
- The runtime cost of `pac-ret+leaf` over `pac-ret` is roughly two
  instructions (`paciasp` + `autiasp`) per leaf function — modest in
  the abstract, occasionally measurable on leaf-heavy workloads. The
  ADR documents this so a downstream consumer reading the suite knows
  the tradeoff exists; it is not a constraint imposed by this ADR.
